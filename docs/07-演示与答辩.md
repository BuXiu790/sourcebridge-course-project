### 6:40–8:00 总结与提问

总结课程亮点：组件化、类型安全、响应式、真实数据库、双层权限、测试和公网部署。

## 2. 技术亮点

- Server Component 读取真实业务数据，Client Component 处理表单与局部状态。
- TypeScript 将 UI 状态、数据库枚举和 API 参数连接起来。
- 服务端角色检查与 PostgreSQL RLS 形成纵深防御。
- 公开 Demo 与真实业务数据物理分离，避免混淆和隐私问题。
- 多步骤表单同时有前端体验校验和服务端最终校验。
- 真实四账号集成脚本不使用 service role，并自动清理临时数据。

## 3. 为什么选择这个技术栈

- Next.js：同时覆盖页面、服务端渲染、路由保护和轻量 API。
- TypeScript：降低字段、角色和状态不一致风险。
- Tailwind：快速建立统一且响应式的 B2B UI。
- Supabase：在课程周期内提供可靠 Auth、PostgreSQL 和 RLS。
- Sites：把同一 Git 课程版本发布为公网可演示站点。

## 4. 为什么使用 PostgreSQL

项目需要多用户、表关系、外键、索引、触发器和数据库级行权限。PostgreSQL、MySQL、SQLite 都是关系型数据库，但 PostgreSQL 的 RLS 与 Supabase Auth 集成最适合本项目。SQLite 更适合单机文件，MySQL 也能实现关系和约束，但需要额外实现认证与行权限服务。

## 5. RLS 如何防止越权

浏览器发出的请求携带当前 Supabase 用户身份。Policy 使用 `auth.uid()` 比较 RFQ 的 `buyer_id`，或检查当前 Profile 是否为 Staff。Buyer 即使手工修改 URL 或直接调用 REST，也只能得到自己的行；角色更新 RPC 内部再次检查 Admin。

## 6. 遇到的问题与解决方法

1. **日期输入 DOM 有值但 React State 未同步**：改用 `onInput` 统一读取原生日期值，并做浏览器回归。
2. **旧测试默认 Supabase 未配置**：抽出纯权限决策函数，分别覆盖未配置 503 和已配置匿名 401。
3. **RLS 更新可能返回 0 行而不是 Error**：测试改为同时验证返回行和数据库最终角色。
4. **Live 测试会留下临时数据**：只在可删除对象上测试写入，并用 `finally` 恢复状态和删除记录。
5. **公开注册与邮件交付**：Course v1.1 为课堂体验暂时关闭邮箱确认并开放 Buyer 注册，同时禁用密码找回；商业化前必须配置 SMTP 并重新开启邮箱确认。

## 7. 教师可能提问与参考回答（15 题）

1. **这是纯前端项目吗？** 以 Next.js 前端框架实践为主，同时用 Route Handler 和 Supabase 完成真实认证与数据层，体现完整工程闭环。
2. **为什么不用普通 React SPA？** Next.js 提供 Server Component、文件路由、服务端鉴权和 API，减少额外后端工程。
3. **为什么不用 MySQL？** MySQL 完全可以存关系数据，但 Supabase PostgreSQL 同时提供 Auth、RLS 和托管能力，更适合课程周期。
4. **PostgreSQL 与 MySQL/SQLite 的共同点？** 都是关系型数据库，使用表、主外键、约束和 SQL。
5. **如何防止 Buyer A 看 Buyer B？** 服务端按会话查询，数据库 RLS 再按 `buyer_id = auth.uid()` 过滤，并做了真实双账号测试。
6. **只隐藏 Admin 按钮够吗？** 不够；页面有 `requireRole`，API 有 `getApiAuth`，数据库写入还有 RLS/RPC 检查。
7. **用户能在注册时成为 Admin 吗？** 不能；数据库 Trigger 固定创建 Buyer，Admin 只能由现有 Admin 的受保护 RPC 调整。
8. **如何保证表单数据正确？** 前端逐步校验改善体验，API 再用独立验证器校验，数据库还有数值、外键和枚举约束。
9. **为什么 Mock 和真实数据分开？** 公开 Demo 必须稳定且不能泄露用户数据；登录后的 Dashboard 只读 Supabase。
10. **报价选择会下单吗？** 不会；真实页面只记录选择偏好，公开 Demo 只更新浏览器状态。
11. **成本估算可靠吗？** 它是明确标注的 Demo 模型，展示计算能力，不是报价或利润承诺。
12. **组件化体现在哪里？** Header、Footer、Auth、Dashboard、Form、Timeline、CommercialAnalysis、Admin、UI 等共 19 个组件。
13. **测试覆盖什么？** 页面渲染、权限双态、输入校验、RLS 契约、敏感信息、真实四角色矩阵和响应式浏览器。
14. **为什么课程版关闭邮箱确认？** 为了让课堂访问者无需依赖默认 SMTP 即可体验 Buyer 流程；页面明确提示邮箱未验证，商业化前必须配置自定义 SMTP、重新开启确认并恢复密码找回。
15. **下一步怎么商业化？** 先配置 SMTP、审计日志和文件 Storage，再评估限流、通知、Amazon/物流 API；支付与真实供应商不属于当前课程范围。
